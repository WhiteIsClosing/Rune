# Templates

按任务类别积累的验收标准模板。Claude 识别出任务属于某个已有类别时，直接套用对应模板，不再重新和用户协商验收标准。

## 命名约定

文件名使用 kebab-case 描述任务类别，一个 `.md` 文件对应一类任务：

- `frontend-component.md`
- `sql-migration.md`
- `api-integration.md`
- `bug-fix.md`
- `refactor.md`

## 模板格式

每个模板必须包含以下五个部分：

```markdown
# <任务类别名>

## 触发条件
什么样的用户请求算是这一类任务？列 2-3 个典型 prompt 特征或关键词，帮 Claude 判断是否套用此模板。

## 验证方式
- 跑哪些命令
- 看哪些输出
- 手动验证步骤（如有）

## 必须覆盖的边界
- 这类任务最容易漏的 2-4 个 case

## 停下来找用户的信号
- 这类任务中哪些判断不该 Claude 自己做

## 范围外
- 这类任务**明确不**涉及的东西
```

## 使用流程

1. Claude 收到任务 → 判断是否匹配某个模板的「触发条件」
2. 如果匹配：直接按该模板输出验收标准，等确认
3. 如果不匹配：按 `rules.md` 走一般流程；完成后若适合沉淀，新增一个模板文件

## 维护原则

- **保持精简**。每个模板只列"漏了会翻车"的关键项，不写穷举列表。
- **发现反模式就更新**。如果某模板导致 Claude 多次判断错误，修改或拆分。
- **不要创建互相重叠的模板**。模糊边界比少一个模板更糟。
