# Rune — 通用项目规范与方法论

本文件是 Rune 母版规范。新建项目时**整段拷贝本文件内容到项目 `CLAUDE.md` 开头**，作为该项目的工程基线。拷贝完归项目所有，项目自治演化；Rune 母版后续更新不向已有项目反向同步。

## 一、定位与使用

Rune 是一份**母版规范**——新建项目时整段拷贝本文件内容到项目 `CLAUDE.md` 开头，作为该项目的工程基线。

- **拷贝后归项目所有**：内容物理上属于项目，项目可自由增删改
- **不再有"跨项目活引用"**：Rune 不通过 submodule / @ 导入注入到项目，所有规则一旦拷贝就是项目自有资产
- **单一规范层**：本文件加上项目自加的规则共同构成项目唯一规范，不再区分 L1/L2

**Rune 自身演化**：母版会持续打磨（见 七、元规则），但更新不反向同步到已有项目。母版改动只影响**未来**新建项目。

**项目专属规则放哪**：项目自加的规则直接追加到项目 `CLAUDE.md` 末尾即可——本文件给出的硬约束（A–F 等）项目可在不冲突的前提下扩展，不必另起文件。

## 二、Claude 协作规则

规则按典型任务的执行时序编号，便于按链条 review。每条统一 4 字段：**何时 / 做什么 / 为什么 / 例外**（无例外时省略）。

### 任务类请求的完整执行链路

五步工程法（需求筛选）→ 规则 1（反向校准）→ 规则 2（任务分类 + 自驱度）→ 规则 3（验收标准）→ 执行 + 规则 4（验证）→ 规则 6 / 7（沉淀）。**规则 8（能量信号）全程监测**。

### 任务接收阶段（你派活 → 我开始之前）

#### 规则 1：反向校准目标与问题

- **何时**：每个任务类请求
- **做什么**：暴露三层翻译，等用户校对再继续

  > 「我理解的：
  >  - 目标 (want)：……（一句话复述，不抄原话）
  >  - 你问的 (ask)：……（具体问题）
  >  - 翻译路径：为什么这个 ask 能服务那个 want
  > 哪一层不对？」

  - 歧义低：单行带过（"want=X / ask=Y / 翻译直接，开始"）
  - 歧义高 / ask 多解读：分行追问，给 2–3 个候选让用户挑（"字面问的 X，但也可能是 Y / Z，哪个？"）

  **不能省略**——省略 = 默默假设 = 高效执行错方向。

  **主动追问 "why" 的触发**：请求和现有规则冲突 / 解法成本明显大于价值 / 用户预设了具体解法（"用 X 解决 Y" — X 不一定最优）/ 反复改但每次方向不同（→ 触发规则 8）。

- **为什么**：错问题的解法做得越快错得越远——30 秒校准避免 30 小时返工
- **例外**：规则 3 的 skip criteria 同时豁免本规则

#### 规则 2：任务分类与自驱度匹配

- **何时**：规则 1 校准通过后，规则 3 之前
- **做什么**：声明任务类型 + 建议自驱度

  | 类型 | 特征 | 自驱度 | 执行 |
  |---|---|---|---|
  | **可验证执行型** | 边界明确 / 可自动化验证 / 有参考实现 / 对错易判断 | 高 | 规则 3+4，完成再来 |
  | **研究 / 品味型** | 架构选择 / 长期一致性 / 美丑判断 / 真新问题 / 验收标准本身不清晰 | **降低** | 分阶段、每阶段对齐后推进，不假装一口气搞定 |

  **声明格式**（验收标准开头加一行）：

  > 「任务类型：可验证执行型 — 高自驱，产出后验收」 / 「研究品味型 — 建议分 N 阶段，X 节点加强 check-in」

- **为什么**：诚实声明"不适合一口气干完"比假装能干好。用户据此决定给多少注意力——这是 1000x 不在错方向烧尽的前提

#### 规则 3：先定验收标准

- **何时**：规则 1、2 后
- **做什么**：输出验收标准四要素，等用户确认后再实现

  1. **验证方式** — 做完如何证明对了（优先自动化命令 / 输出断言；必须手动则写清步骤）
  2. **必须覆盖的边界** — 2–4 个"漏了会翻车"的关键 case，不追求穷举
  3. **停下来找用户的信号** — 哪些情况要回来问（多种合理设计、需求与代码冲突、需引入新依赖 / 改公共接口）
  4. **范围外的事** — 明确**不做**的，防 scope creep

- **例外**（直接动手，不输出验收标准；同时豁免规则 1 的反向校准）：
  - prompt 含 `skip criteria`
  - 对话 / 探索 / 闲聊 / 纯问答（用户在"聊"不是"派活"）
  - 零风险只读操作（读文件、查命令、状态检查）

  **不确定时先输出简短验收标准**——用户可反向跳过，反之不成立。

### 执行阶段（开干 → 声明完成）

#### 规则 4：完成前必须实际验证

- **何时**：声明任务完成之前
- **做什么**：必须有可观测验证证据

  - **代码** — 跑 `ci_local.sh` / 单测 / type check / lint，贴输出
  - **UI / 前端** — E2E 视觉测试或真实浏览器 smoke，不只 `tsc --noEmit`
  - **文档 / 规范** — 逐条回读，确认形式标记到位
  - **外部副作用**（push / merge / 发 PR / 改 CI）— 逐步确认每步结果

  无法实际验证时明确声明"未验证的部分"，不伪装完成。

- **为什么**：类型检查和测试套件只验证**代码正确性**，不验证**功能正确性**。Token 远低于返工成本，不为省 token 跳验证

#### 规则 5：长跑 / 并行任务契约

- **何时**：background agent / worktree 并发 / 单 session 多 PR / 夜间长跑
- **做什么**：

  - **独立性** — 每个 PR / commit 可独立 review 和回滚，不跨 PR 共享未落地状态
  - **低置信度显眼标记** — PR 标题 `[speculative]`、commit `explore:` / `WIP:`、文件头 `// exploratory`。**不要用沉默掩盖不确定**
  - **失败透明** — 探索失败产出短报告（"尝试 X，结论 Y，弃用原因 Z"），不删掉重来装没发生——**失败路径是下次决策的输入**
  - **长跑前对齐"最小验收边界"** — 跑之前敲定"满足哪几条就算有产出"，避免醒来看到 20 个 PR 全在错方向
  - **结束时产出总报告** — 分类列出 done / blocked / rejected，按 review 粒度排序

### 沉淀阶段（任务结束 / review 时）

#### 沉淀的四种载体（规则 6、7 共用）

| 沉淀物性质 | 落点 | 激活方式 |
|---|---|---|
| 任务级重复流程 | `docs/templates/<name>.md` | 同类任务手动套 |
| 约束 / 偏好 / 硬规则 | 项目 `CLAUDE.md` 规则表 | 进 session 上下文自动生效 |
| 可复用操作序列 | `.claude/skills/<name>/` | `/skill-name` 或关键词触发 |
| 自动化触发 / 环境保证 | `.claude/hooks/` + `settings.json` | 事件触发，无需 agent |

**项目内 vs 母版**（与载体选型正交）：当前项目要用 → 项目 `CLAUDE.md`；多个项目都该有 → 同步提议更新 Rune 母版（影响未来新项目）。默认先放当前项目，多次复用后再考虑提母版。

#### 规则 6：完成后主动问沉淀

- **何时**：任务完成且产物**可能再次出现**时（不等用户提）
- **做什么**：主动问沉淀归属

  > 「这次的做法可以沉淀。归在 (a) `docs/templates/` (b) 项目 `CLAUDE.md` 规则表 (c) Rune 母版（已多项目用）(d) `.claude/skills/` (e) `.claude/hooks/` (f) 不沉淀」

  模板形态：简短 markdown（背景 / 步骤 / 验收标准）。

- **为什么**：下次同类任务直接套载体，冷启动成本趋零

#### 规则 7：纠错即沉淀（"教一次，以后不用再教"）

- **何时**：用户指出做错 / 偏了 / 方向不对；纠正做法；表达偏好；质疑选择并给出更优方案
- **做什么**：**当次回复**主动问沉淀归属

  > 「这个纠正值得沉淀。归在 (a) 项目 `CLAUDE.md` 规则表 (b) Rune 母版（若跨项目适用）(c) `.claude/skills/` (d) `.claude/hooks/` 自动拦截 (e) 不沉淀（一次性特例）」

- **为什么**：和规则 6 对偶——规则 6 = 做对沉淀模板，规则 7 = 被纠正沉淀规则。每次 review / 纠错都是"教一次以后不用再教"的机会，浪费 = session 重置从零 = 永远停在 1x。沉淀是突破 1000x 的**必要**条件

### 全程监测

#### 规则 8：能量信号反馈

- **何时**：观察到用户能量信号矛盾——反复修改但方向漂移 / 说不上具体哪不对

  对照：**越改越具体 / 越聊越清楚** 是真目标已立住，不需打断。

- **做什么**：主动打断 + 反馈

  > 「你已经改了 N 版，每次方向不太一样——可能不是解法的问题，是根本要解决的问题还没写清。要不要先用 5 分钟把"如果这事搞定，最终你看到什么"写一段出来？」

- **为什么**：这个观察**不能藏在心里**——哪怕可能让用户感觉被评判。默默继续 = 用 token 帮用户走错路。1000x 不在错方向烧尽的**最后一道防线**

## 三、工程方法论

### 第一性原理

所有决策追溯到第一性原理，禁止"别人这么做所以我也这么做"的类比。**P1（域第一性原理）由项目自补**；以下 P2–P5 通用：

- **P2 数据与计算** — 每步计算都在改变度量。选最直接的计算；每步变换必须有明确目的（平滑→消噪、归一化→跨尺度可比）；副作用（滞后、丢失量级）须在注释中声明
- **P3 架构与设计** — 软件结构 = 问题结构。模块边界 = 业务概念边界；依赖方向 = 信息流方向；复杂度匹配业务复杂度
- **P4 代码实现** — 代码是业务逻辑的精确编码。每个函数能一句话描述目的；实现选择有业务理由；不写防御性冗余
- **P5 测试与验证** — 单测用手算 expected value；集成测试验 pipeline；测试和预期矛盾时质疑实现而非测试

### 五步工程法

按序执行，不可跳步。核心精神：**最好的代码是不需要写的代码**。

1. **质疑需求** — "不做会怎样？" 默认立场是不做
2. **删除** — 大胆删，优先于优化
3. **简化** — 更少步骤、状态、配置项
4. **加速** — 缩短想法到验证的反馈循环
5. **自动化** — 前四步完成后才自动化

## 四、工作流细则

**会话初始化** — `git fetch origin main && git pull origin main`。

**分支与 PR** — 命名 `claude/<topic>-<slug>`；**禁直推 `main`**；PR 严格单一主题，无关清理另开 PR。

**PR test plan 由 agent 填写完整** — 不留 TODO / "manual test" / 空白让用户填。Test plan 是规则 4 验证产物的对外形式：列实际跑过的命令、看到的输出、覆盖的 case；未验证的部分必须明示标注（"功能 X 未在浏览器实测"），不伪装完成。模板：

```markdown
## Summary
- <1–3 bullets：改了什么 / 为什么>

## Test plan
- [x] <实际跑过的命令> — <关键输出 / 断言>
- [x] <覆盖的边界 case>
- [ ] <未跑的部分明示原因，例："X 未在浏览器实测">
```

**大任务拆分**（>150 行触发，不可豁免）：

1. 先列步骤清单（每步 ≤100 行增量），用户确认后再执行
2. 每完成一步立即汇报
3. **单次工具调用写入硬上限 200 行**；150–200 行建议拆分但不强制
4. **中间状态必须合法**：不留未闭合代码块 / 残缺章节

**文件加载白名单** — 主动只读 `.md` `.py`（或项目主语言）`.toml` `.yml` `.yaml` `.sh`。其他后缀（`.csv` / `.parquet` / 二进制 / 截图）非用户明确要求不读。

## 五、硬约束最小公约数（A–F）

跨项目共性约束（母版）。新项目继承后**可加更严，不可弱化任一条**。本节编号保留；项目专属规则（如前瞻偏差、框架隔离等）由项目在本节后接续追加 A2 / A3...

### 审查严重级（A–F 表格"级别"列的图例）

四档制，贯穿 PR review、`/review` 输出、本地 `ci_local.sh`：

| 级别 | 含义 | 典型场景 |
|------|------|----------|
| **CRITICAL** | 阻塞合并，必须当场修 | DAG 循环依赖（C5）、前瞻偏差、时区错位（A-TZ）、核心计算静默失效 |
| **HIGH** | 本 PR 必修 | 入口不校验、静默吞异常、幂等破坏、类型不完整、覆盖率不足、CI 失败 |
| **MEDIUM** | 本 PR 内修复 | 日志规范、冗余代码、文件 / 函数超限、命名不清晰、文档-代码不同步 |
| **LOW** | 可建议，不阻塞 | 版本上界缺失、风格偏好、非关键文案 |

### A. 数据完整性

| 编号 | 规则 | 级别 |
|---|---|---|
| A1 | 公共 API / 外部数据入口必须校验（空值、类型、形状、范围） | HIGH |
| A-IMM | 入参对象不可变：禁止就地修改传入对象；修改前 `.copy()`；内部新建对象可原地改。**具体容器和标志由项目指定** | HIGH |
| A-BND | 数值边界处理：NaN / inf / 空容器 / 长度不足 / 时序间隙 | HIGH |
| A-TZ | 金融计算时区一律 UTC：内部存储 / 计算用 tz-aware UTC datetime，**禁 naive datetime 参与金融计算**；外部入口（行情、订单、撮合、回测起止）必须显式标注源时区，转 UTC 后再进入计算；配置中时间字段必须显式标注时区（如 `"America/New_York"` / `"UTC"`），禁裸字符串；跨市场 / 跨数据源聚合必须先对齐 UTC。展示 / 日志 / CLI 输出允许转交易所或本地时区 | **CRITICAL** |

### B. 可观测性

| 编号 | 规则 | 级别 |
|---|---|---|
| B1 | 禁止静默吞异常：`except: pass` 绝对禁止；捕获必须有行动（日志 / 重抛 / 降级并告警） | HIGH |
| B2 | 库代码用 `logging.getLogger(__name__)`，禁 `print()`；CLI 入口允许 `print()` | MEDIUM |
| B3 | 禁止静默降级：可选输入退化时**必须** `logging.warning` + 阈值配置化 | HIGH |

### C. 代码结构

| 编号 | 规则 | 级别 |
|---|---|---|
| C1 | 计算幂等：相同输入 → 相同输出 | HIGH |
| C2 | 完备性：参数必被用；Enum / 分支穷尽且 default 抛异常；dataclass 全填；配置字段必被消费，计算参数必来自配置 | HIGH |
| C3 | 禁止冗余代码：死代码、注释代码**直接删**，不留"以后可能用" | MEDIUM |
| C4 | 模块粒度：非测试文件 ≤800 行（纯配置类文件 ≤1000，仅含 `@dataclass(frozen=True)` 与工厂函数），函数 ≤120 行；测试文件与纯声明式注册表文件不受限 | MEDIUM |
| C5 | DAG 架构：禁循环依赖；上游禁 import 下游；架构文档**必须**含 DAG 图（ASCII / Mermaid） | **CRITICAL** |
| C6 | 命名自解释：禁 `tmp` / `data` / `result` / `val` / `process()` 等无信息命名 | MEDIUM |

### D. 配置与参数

| 编号 | 规则 | 级别 |
|---|---|---|
| D1 | 配置类型化：`@dataclass` + 类型标注（按需 `frozen` / `slots`） | HIGH |
| D2 | 业务参数（窗口 / 阈值 / 手数）必须配置化，禁魔法数 | HIGH |

### E. 工程规范

| 编号 | 规则 | 级别 |
|---|---|---|
| E1 | 敏感信息禁硬编码：密钥 / token 走环境变量或 `.gitignore`；启用 `detect-secrets` 扫描 | HIGH |
| E2 | Agent 调用用最新 Opus 模型 | HIGH |
| E3 | 回复使用中文（代码标识符 / commit / PR 标题除外） | MEDIUM |
| E4 | 运行时依赖指定版本上界（如 `numpy>=2.0,<3.0`） | LOW |

### F. 测试、类型与文档

| 编号 | 规则 | 级别 |
|---|---|---|
| F1 | 测试三层：**单元**（纯函数 + 真实数据，**禁 mock 内部模块**）→ **集成**（可 mock 外部 I/O）→ **端到端**。覆盖率 `fail_under=60`（项目可提高） | HIGH |
| F2 | 代码-文档自底向上同步：代码 → `docs/` →（影响架构时）`CLAUDE.md`。`docs/` 禁记录行号 / 私有符号（`_` 前缀），默认值不重复维护 | MEDIUM |
| F3 | 类型标注严格模式：`mypy --strict` 或等价配置 | HIGH |
| F4 | CI/CD 护栏：合并 `main` 的 PR 必须过全部 CI job；**本地 `ci_local.sh` 硬上限 90 秒**（软上限 45 秒）——超即 fail，防 gate 腐化 | HIGH |

### 规则级可观测性

每条硬约束必须**可观测地合规**。来源按优先级：(1) 自动化工具（ruff / mypy / import-linter / pytest / 自定义脚本）；(2) 项目 `CLAUDE.md` 指定的形式标记；(3) 响应末尾"规则合规清单"（兜底，不推荐）。

**必须维护"规则级约束表"**：为本节每条规则绑定自动化工具或形式标记。新增 / 修改规则时同一 PR 必须更新——否则"规则生效但合规不可观测" = 静默失效。

**绑定示例（A-TZ）**：自动化用 `mypy --strict` 校验 `datetime` 字段为 tz-aware + 自定义脚本扫描 naive datetime 字面量（如 `datetime(...)` / `datetime.now()` / `pd.Timestamp(...)` 缺 tz 参数）；形式标记用配置 dataclass 时间字段必须含 `tz: str` 或类型为 tz-aware datetime。

## 六、新项目引入 Rune

新建项目时一次性引入，从此项目自治。

### 6 步引入

1. 新建仓库（`git init` / `gh repo create`）
2. 项目根建 `CLAUDE.md`
3. 整段拷贝本文件（从 `# Rune` 到末尾，含反模式）到 `CLAUDE.md` 开头
4. 在末尾追加项目专属内容：项目简介 / 第一性原理 P1 / 项目专属硬约束（接 A2 / A3 / B4...）/ 规则级约束表（见 五.规则级可观测性）。建议结构：H2 `## 九、<项目名> 项目专属` + H3 `9.1 项目简介 / 9.2 P1 / 9.3 硬约束接续 / 9.4 规则级约束表`
5. 删除"## 六、新项目引入 Rune"整节——这节是 Rune 自己的引入指引，对项目无意义
6. **本地化检查表**：替换母版里 Rune 自称视角的措辞为项目第一人称，避免"项目 `CLAUDE.md` 自称是 Rune 母版"的 dissonance。可手动逐项 Edit，也可填 `.claude/skills/sync-rune-master/examples/adoption-localization-spec.json` 里的 `<项目名>` / `<commit>` / `<date>` 占位符后跑 `sync.py` 自动化：

   | 位置 | 改前（Rune 自称） | 改后（项目自治） |
   |---|---|---|
   | L1 标题 | `# Rune — 通用项目规范与方法论` | `# <项目名> — 工程基线` |
   | 开头描述 | "本文件是 Rune 母版规范……" | "本文件是 `<项目名>` 的工程基线，由 Rune 母版（commit `<sha>`，`<date>`）整段拷贝引入……" |

   | §一 自称 | "Rune 是一份**母版规范**——新建项目时整段拷贝……" | "本文件是 `<项目名>` 的工程基线，从 Rune 母版整段拷贝（commit `<sha>`，`<date>`），引入后归项目自治" |
   | §一 "Rune 自身演化" 段 | "**Rune 自身演化**：母版会持续打磨……" | "**与 Rune 母版的关系**：母版会在 Rune 仓持续打磨；非 CRITICAL 改动不反向同步，要采纳靠手动 diff + cherry-pick；CRITICAL 级新规则由 Rune 通过 `sync-rune-master` skill 主动开 PR 到本仓（详 §七）" |

### 引入后

- 项目 `CLAUDE.md` 完整自洽，session 启动 Claude Code 自动加载；没有 submodule / hook / @ 导入，全是本地文件
- 母版后续更新**不**反向同步；要同步自己手动 diff + cherry-pick

参考实现：SIGIL、Vates、Scry（旧 submodule 模型，新项目按上面 6 步走）；Stargazer 是按 6 步走引入并完成本地化的首个项目，可直接对照其 PR diff。

## 七、元规则：持续打磨 Rune

协作中发现以下情况，**主动提议**更新 Rune 母版（添、删、改都包括）：

- 母版规则写得不清楚 / 会误判 → 改本文件
- 母版规则在实际项目里几乎都被覆写或忽略 → 提议简化或删除
- 项目专属规则在多个项目都重复出现 → 提议提升到 Rune 母版
- 某个模板 / 约束从未被执行或复用 → 提议删除
- 新工作流模式值得固化 → 提议新增规则

Rune 是活文档，不是刻在石头上的。**该加就加，该砍就砍**——每条规则都必须能在实际项目里观测到收益，否则它就是噪音。

**重要（默认）**：母版改动只影响**未来**新建项目。已有项目要不要同步是各项目自己的事；Rune 不主动推送。

**例外（CRITICAL 级回溯传播）**：CRITICAL 级新规则（如 A-TZ）破坏正确性而非偏好——这种规则建议**主动回溯**到所有已有项目，避免"母版有规、项目无规"的合规漂移。流程见 `.claude/skills/sync-rune-master/`：写一份 patch spec（`(old, new)` 字符串对 + 目标项目列表）→ 跑 `sync.py` → 自动 fetch / patch / push / open PR 到所有姐妹项目。HIGH 及以下级别**不**建议自动回溯，留给项目自决。

## 八、反模式

**引入与演化层面：**

- ❌ 把项目专属规则提交到 Rune 母版（应留在项目 `CLAUDE.md` 自己加） → 一.单一规范层
- ❌ 引入时只拷贝部分章节，遗漏 A–F 或反模式 → 六.步骤 3
- ❌ 引入后跑回 Rune 母版改规则（应改项目自己的 `CLAUDE.md`） → 一.拷贝后归项目所有
- ❌ 引入后没追加项目 P1 / 项目硬约束 / 规则级约束表 → 六.步骤 4
- ❌ 引入后没删"## 六、新项目引入 Rune"节（保留无意义） → 六.步骤 5
- ❌ 引入后跳过本地化（标题 / §一 / §七 / §八 仍以 Rune 自称，"项目 CLAUDE.md 自称是 Rune 母版"） → 六.步骤 6
- ❌ Rune 母版改了，期望已有项目自动同步（不会同步，要手动 diff） → 一.不再有跨项目活引用

**工程层面：**

- ❌ 类比思维：因为某框架这么做所以我也——必须追溯第一性原理 → 三.第一性原理
- ❌ 跳过五步工程法，还没质疑需求就动手 → 三.五步工程法
- ❌ 防御性冗余：在上游已保证的地方再校验 → 三.P4
- ❌ 金融计算用 naive datetime / 跨时区裸字符串聚合 / 配置裸时间字面量不标 tz → 五.A-TZ
- ❌ 惯性变换：无明确目的的归一化 / 平滑 / 填充 → 三.P2
- ❌ 靠感觉说"做完"：无自动化验证 / 无 E2E 截图 / 无测试证据 → 二.规则 4
- ❌ 用"以后可能用"保留死代码 / 注释代码 → 五.C3
- ❌ mock 内部模块跑单测（等于测空气）→ 五.F1
- ❌ 文档里写代码行号 / 私有符号 / 重复 dataclass 默认值 → 五.F2
- ❌ 本地 CI 脚本超 90 秒（gate 会腐化，没人愿意等）→ 五.F4
