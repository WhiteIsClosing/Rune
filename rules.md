# Claude 工作流规则（来自 Rune）

以下规则由 SessionStart hook 注入到当前 session 上下文，对所有接入 Rune 的项目生效。

---

## Rune 的结构

Rune 分两层：

- **L1 通用层（`rules.md`，本文件）**：与具体项目无关的 agentic 协作规则，服务所有接入 Rune 的体系。
- **L2 项目层（`<project>/`）**：单个项目（即一个 repository）自身的规范和约束。按项目名建子目录，内部结构由项目自行组织（例如某个量化策略项目下的风控约束、回测对齐、精度要求、时区约定、常用模板等）。

### 判断归属的决策树

遇到"这条规则 / 知识 / 模板该放哪"时：

1. **跨项目通用** → L1（`rules.md`）
2. **只属于某个项目** → L2（该项目的 `<project>/`）
3. **拿不准** → 先放 L2；观察多次使用后再决定是否升级到 L1

---

## 规则 1：先定验收标准，再动手

收到**任务类请求**时，**不要立即开始实现**。先输出一份验收标准，等用户确认或修改后再执行。

验收标准包含四个部分：

### 1. 验证方式
做完后如何证明对了？优先可自动化：
- 跑什么命令（测试、lint、type check、构建等）
- 看什么输出或断言
- 如果必须手动验证，写清具体步骤

### 2. 必须覆盖的边界
列 2-4 个最关键的 case。不追求穷举，只列"漏了会翻车"的。

### 3. 停下来找用户的信号
什么情况应该回来问，而不是自己猜？例如：
- 遇到多种合理的设计选项
- 发现原需求和现有代码冲突
- 需要引入新依赖 / 改动公共接口

### 4. 范围外的事
明确**不做**的东西，防 scope creep。

---

## 规则 2：不走验收标准的例外

以下情况直接动手，**不要**先输出验收标准：

- Prompt 中包含字符串 `skip criteria`
- 对话、探索、闲聊、概念讨论、纯问答类请求（用户在"聊"不是"派活"）
- 零风险只读操作：读文件、查询命令、跑状态检查等

判断不确定时，倾向于**先输出简短的验收标准**——用户总可以说"skip criteria"让你跳过，反之不成立。

---

## 规则 3：每完成一类任务，沉淀模板

完成任务后，如果该任务属于一个**可能再次出现的类别**（前端组件、SQL 迁移、API 接入、bug 修复等），**主动发起一次沉淀询问**——不要等用户提。

### 沉淀的落点

和规则/知识同样按归属决策树判断：

- **只在当前项目会再次出现** → `<project>/` 下（例如 `<project>/templates/<name>.md`）
- **多个项目都会用到** → 提升为 L1 的通用模板

默认先放项目层，至少在**第二个项目**实际用到了再考虑提升，别提前抽象。

### 沉淀的提问格式

任务完成时主动问一次：

> 「这次的做法可以沉淀。归在：
>   (a) 当前项目 `<project>/`
>   (b) 提升到 L1 通用模板（仅当已在多个项目使用）
>   (c) 不沉淀」

模板本身是简短 markdown：一段背景、一段步骤、一段验收标准。下次遇到同类任务直接套模板，冷启动成本趋近于零。

---

## 元规则：持续打磨 Rune 整体

协作中发现以下情况，都**主动提议**更新 Rune（不只是 `rules.md`）：

- 某条 L1 规则写得不清楚 / 会误判 → 改 `rules.md`
- 发现一条项目级别的规范或约束还没文档化 → 新增或更新 `<project>/` 下对应文件
- 某条 L1 规则其实只在一个项目用得到 → 提议下沉到该项目的 L2
- 某条 L2 规则 / 模板在多个项目都出现 → 提议提升到 L1
- 某个模板从未被复用 → 提议删除
- 新工作流模式值得固化 → 提议新增规则或模板

Rune 是活文档，不是刻在石头上的。该加就加，该砍就砍。
