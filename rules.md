# Rune — 通用项目规范与方法论

本文件是跨项目的 L1 规范与方法论单一事实源。通过 `.rune/` submodule 挂载到消费项目，在消费项目的 `CLAUDE.md` 用 `@.rune/rules.md` 导入到 session 上下文。

---

## 一、定位与分层

Rune 分两层：

- **L1 通用层（本文件）** — 跨项目的 agentic 协作规则 + 工程共性约束 + 方法论。项目无关。
- **L2 项目层（消费项目自身的 `CLAUDE.md` / `docs/`）** — 单个项目专属的硬约束、域知识、参考实现。

### 冲突时的优先级

L2 > L1。L1 是协作风格与共性，L2 是工程正确性。冲突时以 L2 为准，但 **L2 只能加强、不能弱化 L1 的硬约束**。

### 判断归属的决策树

遇到"这条规则 / 知识 / 模板该放哪"时：

1. **跨项目通用** → L1（本文件）
2. **只属于某个项目** → L2（该项目的 `CLAUDE.md` / `docs/`）
3. **拿不准** → 先放 L2；观察多次使用后再提议升级到 L1

## 二、Claude 协作规则

### 规则 1：先定验收标准，再动手

收到**任务类请求**时，**不要立即开始实现**。先输出一份验收标准，等用户确认或修改后再执行。

验收标准包含四个部分：

**1. 验证方式**：做完后如何证明对了？优先可自动化——跑什么命令（测试、lint、type check、构建等）、看什么输出或断言。必须手动验证时写清具体步骤。

**2. 必须覆盖的边界**：列 2–4 个最关键的 case。不追求穷举，只列"漏了会翻车"的。

**3. 停下来找用户的信号**：什么情况应该回来问，而不是自己猜？例如遇到多种合理设计选项、原需求与现有代码冲突、需要引入新依赖 / 改公共接口。

**4. 范围外的事**：明确**不做**的东西，防 scope creep。

### 规则 2：不走验收标准的例外

以下情况直接动手，**不要**先输出验收标准：

- Prompt 中包含字符串 `skip criteria`
- 对话、探索、闲聊、概念讨论、纯问答类请求（用户在"聊"不是"派活"）
- 零风险只读操作：读文件、查询命令、跑状态检查等

判断不确定时，倾向于**先输出简短的验收标准**——用户总可以说"skip criteria"让你跳过，反之不成立。

### 规则 3：每完成一类任务，沉淀到合适载体

完成任务后，如果该任务产生了**可能再次出现**的产物（流程、规则、操作序列、自动化触发），**主动发起一次沉淀询问**——不要等用户提。

**沉淀的四种载体**（选型决策树）：

| 沉淀物性质 | 落点 | 激活方式 |
|---|---|---|
| 任务级重复流程（前端组件、SQL 迁移、API 接入模板等） | `docs/templates/<name>.md` | 下次同类任务手动套用 |
| 约束 / 风格偏好 / 硬规则 | L2 `CLAUDE.md` 规则表（项目专属）或 L1 `rules.md`（跨项目） | 进 session 上下文自动生效 |
| 可复用操作序列（多步骤组合、调试 playbook） | `.claude/skills/<name>/` | 用户 `/skill-name` 或关键词触发 |
| 自动化触发 / 环境保证（格式化、预检、session 初始化） | `.claude/hooks/` + `.claude/settings.json` | 事件触发，无需 agent 参与 |

**跨项目 / 项目内的二次判断**（与载体选型正交）：

- **只在当前项目会再次出现** → 项目层（`docs/templates/` / L2 `CLAUDE.md` / 项目 `.claude/`）
- **多个项目都会用到** → L1（Rune `rules.md`）

默认先放项目层，至少在**第二个项目**实际用到了再考虑提升到 L1，别提前抽象。

**沉淀的提问格式**：任务完成时主动问一次：

> 「这次的做法可以沉淀。归在：
>   (a) `docs/templates/` 模板
>   (b) L2 `CLAUDE.md` 规则表（项目专属约束）
>   (c) L1 `rules.md` 通用规则（仅当已在多个项目使用）
>   (d) `.claude/skills/` 可复用操作
>   (e) `.claude/hooks/` 自动化
>   (f) 不沉淀」

模板本身是简短 markdown：一段背景、一段步骤、一段验收标准。下次遇到同类任务直接套载体，冷启动成本趋近于零。

### 规则 4：声明完成前必须实际验证

**不能靠"感觉"说做完了**。声明任务完成前，必须有可观测的验证证据：

- **代码类任务**：跑 `ci_local.sh` / 单测 / type check / lint，贴通过截图或输出
- **UI / 前端改动**：跑 E2E 视觉测试或真实浏览器 smoke，不能只依赖 `tsc --noEmit`
- **文档 / 规范类任务**：逐条回读产物，确认形式标记到位
- **外部副作用任务**（push、merge、发 PR、改 CI）：逐步确认每一步结果

类型检查和测试套件**只验证代码正确性**，不验证**功能正确性**——能不能跑通和功能对不对是两回事。无法实际验证时，向用户明确声明"未验证的部分"，不要伪装完成。

**Token 不是主要成本**：跑测试 / lint / type check 的 token 远低于返工。不为省 token 跳过验证。

### 规则 5：纠错即沉淀（"教我一次，以后不用再教"）

和规则 3 对偶——规则 3 是**做对了沉淀模板**，规则 5 是**被纠正时沉淀规则**。

**触发条件**（任一满足即激活）：

- 用户指出你做错了、偏了、方向不对
- 用户纠正你的做法（"下次不要 X"、"这里应该用 Y"、"为什么你总是 Z"）
- 用户表达偏好 / 风格约束
- 用户质疑你的某个选择并给出更优方案

**动作**：**不等用户提**，在当次回复里主动问：

> 「这个纠正值得沉淀，避免下次重犯。归在：
>   (a) L2 `CLAUDE.md` 规则表（项目专属）
>   (b) L1 `rules.md` 规则（若跨项目适用）
>   (c) `.claude/skills/` 可复用操作
>   (d) `.claude/hooks/` 自动拦截
>   (e) 不沉淀（一次性特例）」

**原则**：每次 review / 纠错都是一次"教一次，以后不用再教"的机会——浪费这些机会 = 每次 session 重置从零开始 = 永远停在 1x。沉淀到持久化载体是突破 1000x 的必要条件。

### 规则 6：任务分类与自驱度匹配

**收到任务时，在输出验收标准（规则 1）之前，先分类并声明**：

| 任务类型 | 特征 | 自驱度 | 执行模式 |
|---|---|---|---|
| **可验证执行型** | 有明确边界 / 能自动化验证 / 已有参考实现 / 结果对错易判断 | 高 | 按规则 1+4 走，完成再来 |
| **研究 / 品味型** | 架构选择 / 长期一致性 / 美丑判断 / 真正的新问题 / 验收标准本身不清晰 | **降低**，多次 check-in | 分阶段产出、每阶段先对齐再推进，不假装一口气搞定 |

**声明格式**（在验收标准开头加一行）：

> 「任务类型：**可验证执行型** — 高自驱，产出后你验收」
> 或
> 「任务类型：**研究/品味型** — 建议分 N 阶段，每阶段先对齐再推进；我在架构一致性 / 品味判断上会飘，请在 X 节点加强 check-in」

**原则**：诚实声明"我不适合一口气干完这个"比假装能干好。用户据此决定给多少注意力——**这是 1000x 不会在错方向上烧尽的前提**。

### 规则 7：长跑 / 并行任务契约

适用于：background agent、worktree 并发、单 session 产出多个 PR、夜间长跑。

- **独立性**：每个 PR / commit 可独立 review、独立回滚；不跨 PR 共享未落地的状态
- **低置信度必须显眼标记**：不确定方向的产物加前缀——
    - PR 标题：`[speculative]` / `[探索]`
    - commit message 首行：`WIP:` 或 `explore:`
    - 文件头注释：`// exploratory — 不是最终方向`
  **不要用沉默掩盖不确定**
- **失败透明**：探索失败要产出短报告（"尝试了 X，结论 Y，弃用原因 Z"），而不是删掉重来装没发生——**失败路径本身是下次决策的输入**
- **夜间长跑前对齐"最小验收边界"**：跑之前先和用户敲定"满足哪几条就算有产出"，避免醒来看到 20 个 PR 全在错方向
- **并行任务产出汇总**：长跑结束时产出总报告，分类列出 "done / blocked / rejected"，按可 review 粒度排序

## 三、工程方法论

### 第一性原理

所有开发决策必须能追溯到第一性原理。禁止"别人这么做所以我也这么做"的类比思维。以下 P2–P5 是通用部分；**P1（业务/域第一性原理）由各项目在 L2 补齐**。

**P2. 数据与计算** — 数据是对现实的度量，每一步计算都在改变这个度量。

1. **选择最直接的计算**：能用一步向量化操作就不拆成循环
2. **每步变换必须有明确目的**：平滑→消噪、归一化→跨尺度可比。禁止惯性操作
3. **变换副作用必须被认知**：平滑引入滞后、归一化丢失绝对量级。副作用须在注释中声明

**P3. 架构与设计** — 软件结构是问题结构的自然映射。

1. **模块边界 = 业务概念边界**：每个模块须能用一个业务名词命名
2. **依赖方向 = 信息流方向**：逆向依赖说明抽象有问题
3. **复杂度必须有对应的业务复杂度**：不要为简单逻辑套设计模式

**P4. 代码实现** — 代码是业务逻辑的精确编码，不是更多，也不是更少。

1. **每个函数必须能用一句话描述目的**：描述不出来说明职责不清晰
2. **实现选择必须有业务理由**：为什么用 A 不用 B？答案须追溯到业务逻辑
3. **不写防御性冗余**：上游已保证的，下游不再检查

**P5. 测试与验证** — 测试是对假设的科学检验。

1. **单元测试验证计算正确性**：用手算结果做 expected value
2. **集成测试验证 pipeline 正确性**：各模块组合后行为符合预期
3. **测试结果与预期矛盾时，质疑实现而非测试数据**

### 五步工程法

所有功能开发**必须**按以下顺序执行，不可跳步。核心精神：**最好的代码是不需要写的代码**。

1. **质疑需求**：每个功能引入前回答"不做这个会怎样"。默认立场是"不做"
2. **删除**：大胆删除不必要的部分。删除优先于优化——不要优化一个不该存在的东西
3. **简化**：更少步骤、更少中间状态、更少配置项
4. **加速**：缩短从想法到验证的反馈循环
5. **自动化**：以上四步完成后，最后才自动化

### 五步工程法与规则 1、6 的协同

任务类请求的执行链路：

1. **五步工程法**（需求筛选层）：先过"质疑 → 删除 → 简化 → 加速 → 自动化"。需求被砍掉就直接结束
2. **规则 6**（任务分类层）：需求立住后，声明任务类型（可验证执行型 / 研究品味型）与自驱度
3. **规则 1**（交付契约层）：再输出验收标准四要素
4. **执行 + 规则 4**（验证层）：实现并给出验证证据
5. **规则 3 / 规则 5**（沉淀层）：做对了沉淀模板，被纠正了沉淀规则

## 四、工作流细则

### 会话初始化

每次会话开始时，先同步主分支：

```bash
git fetch origin main && git pull origin main
# 有 submodule 的项目再加一步
git submodule update --init --recursive
```

### 分支与 PR

- 分支命名：`claude/<topic>-<slug>`（例：`claude/review-project-standards-GFrQZ`）
- **禁止直推 `main`**，所有改动走 PR
- PR 内容严格对应单一主题；无关清理另开 PR

### 大任务拆分规范

当单次变更的**新增 + 修改行数合计超过 150 行**，触发拆分流程（不可豁免）：

1. **先列计划**：列出步骤清单（每步 ≤100 行增量），供用户确认后再执行
2. **逐步执行**：每完成一个小步骤立即汇报进度
3. **单次写入硬上限 200 行**：单次工具调用写入绝对不得超过 200 行；150–200 行之间建议拆分但不强制
4. **中间状态必须合法**：每步完成后文件必须处于语法 / 结构合法状态，不得留下未闭合代码块或残缺章节

### 文件加载白名单

理解工程时，**只主动加载以下后缀**：

- `.md` — 文档
- `.py` / 项目主语言源码
- `.toml` — 项目配置
- `.yml` / `.yaml` — CI / 配置
- `.sh` — 工程脚本

其他后缀（`.csv`、`.parquet`、二进制、截图等）**不主动读**，除非用户明确要求或任务直接涉及。

## 五、硬约束最小公约数（A–F）

以下为跨项目共性约束。**L2 可追加更严格的项目特定规则，不可弱化本节任一条**。

域特定规则（如前瞻偏差、数组不可变、时区、框架隔离等）由项目 L2 定义并编号（可直接在本节后追加 A2 / A3 / A4...），**本节编号为 L1 保留**。

### A. 数据完整性

| 编号 | 规则 | 级别 |
|------|------|------|
| A1 | 公共 API / 外部数据入口必须校验（空值、类型、形状、范围） | HIGH |
| A-IMM | 入参对象（数组、DataFrame 等）不可变：禁止就地修改传入对象；修改前 `.copy()`；函数内部新建的对象可原地修改。**具体形式（哪种容器、什么标志）由 L2 指定** | HIGH |
| A-BND | 数值计算必须处理边界：NaN、inf、空容器、长度不足、时序间隙 | HIGH |

### B. 可观测性

| 编号 | 规则 | 级别 |
|------|------|------|
| B1 | 禁止静默吞异常：`except: pass` 绝对禁止；捕获必须有行动（日志、重抛、降级并告警） | HIGH |
| B2 | 库代码必须使用 `logging.getLogger(__name__)`，禁止 `print()`；CLI 入口脚本允许 `print()` | MEDIUM |
| B3 | 禁止静默降级：函数依赖可选输入退化时，**必须**发 `logging.warning` + 降级阈值配置化 | HIGH |

### C. 代码结构

| 编号 | 规则 | 级别 |
|------|------|------|
| C1 | 计算逻辑必须幂等：相同输入 → 相同输出 | HIGH |
| C2 | 完备性：参数必被使用；Enum / 分支必须穷尽且 default 抛异常；dataclass 字段在所有构造路径被填充；配置字段必被消费，计算参数必来自配置 | HIGH |
| C3 | 禁止冗余代码：死代码、注释掉的代码块**直接删除**，不留"以后可能用"的注释 | MEDIUM |
| C4 | 模块粒度：非测试文件 ≤500 行（纯配置文件 ≤800 行），函数体 ≤80 行 | MEDIUM |
| C5 | DAG 架构：禁止循环依赖；上游禁止 import 下游。架构设计文档**必须**包含 DAG 依赖关系图（ASCII / Mermaid） | **CRITICAL** |
| C6 | 命名自解释：禁止 `tmp`/`data`/`result`/`val`/`process()` 等无信息命名 | MEDIUM |

### D. 配置与参数

| 编号 | 规则 | 级别 |
|------|------|------|
| D1 | 配置必须类型化：`@dataclass` + 类型标注（各项目按需加 `frozen`/`slots`） | HIGH |
| D2 | 业务参数必须配置化：窗口 / 阈值 / 手数等不允许散落在代码中作为魔法数 | HIGH |

### E. 工程规范

| 编号 | 规则 | 级别 |
|------|------|------|
| E1 | 敏感信息禁止硬编码：密钥 / token / 口令走环境变量或 `.gitignore` 文件；启用 `detect-secrets` 类扫描 | HIGH |
| E2 | Agent 调用必须使用最新 Opus 模型（具体 ID 以当前 Rune 定义为准） | HIGH |
| E3 | 所有回复使用中文（代码标识符、commit message、PR 标题除外） | MEDIUM |
| E4 | 运行时依赖须指定版本上界（如 `numpy>=2.0,<3.0`） | LOW |

### F. 测试、类型与文档

| 编号 | 规则 | 级别 |
|------|------|------|
| F1 | 测试三层分层：**单元**（纯函数 + 真实数据，**禁止 mock 内部模块**）→ **集成**（可 mock 外部 I/O）→ **端到端**（完整 pipeline）。覆盖率 `fail_under=60`（项目可提高） | HIGH |
| F2 | 代码-文档自底向上同步：代码 → `docs/` 模块文档 →（影响架构时）`CLAUDE.md`。`docs/` 中**禁止记录代码行号**、**禁止记录私有符号**（`_` 前缀），默认值不重复维护（以源码为唯一事实源） | MEDIUM |
| F3 | 类型标注严格模式：`mypy --strict` 或等价严格配置（`disallow_untyped_defs` / `check_untyped_defs` / `no_implicit_optional` / `strict_equality` / `warn_return_any` / `warn_unused_configs`） | HIGH |
| F4 | CI/CD 自动化护栏：所有合并到 `main` 的 PR 必须通过全部 CI job；**本地 `scripts/ci_local.sh` 硬上限 90 秒**（软上限 45 秒触发警告）——超过即 fail，防止 gate 腐化 | HIGH |

### 规则级可观测性

**原则**：每条 L1 硬约束必须在实际产物里**可观测地合规**——审查者能一眼看出"这条规则有没有被遵守"。

**合规可观测性来源**（按优先级）：

1. **自动化工具**：ruff / mypy / import-linter / pytest / 自定义检查脚本（优先）
2. **L2 指定的形式标记**：响应/文档的规定段落结构
3. **响应末尾"规则合规清单"**（兜底，不推荐）

消费项目的 `CLAUDE.md` 应维护一张"**规则级约束表**"，为本节每条规则绑定具体的自动化工具或形式标记。引入新规则时，同一 PR 必须更新该表——否则属于"规则生效但合规不可观测"，等同静默失效。

---

## 六、审查严重级

四档制，贯穿 PR review、`/review` 输出、本地 `ci_local.sh`：

| 级别 | 含义 | 典型场景 |
|------|------|----------|
| **CRITICAL** | 阻塞合并，必须当场修 | DAG 循环依赖（C5）、前瞻偏差、核心计算静默失效 |
| **HIGH** | 本 PR 必修 | 入口不校验、静默吞异常、幂等破坏、类型不完整、覆盖率不足、CI 失败 |
| **MEDIUM** | 本 PR 内修复 | 日志规范、冗余代码、文件 / 函数超限、命名不清晰、文档-代码不同步 |
| **LOW** | 可建议，不阻塞 | 版本上界缺失、风格偏好、非关键文案 |

---

## 七、元规则：持续打磨 Rune

协作中发现以下情况，都**主动提议**更新 Rune（不只是添新规则，也包括删和改）：

- 某条 L1 规则写得不清楚 / 会误判 → 改本文件
- 某条 L1 规则其实只在一个项目用得到 → 提议下沉到该项目的 L2
- 某条项目级规则在多个项目都出现 → 提议提升到 L1
- 某个模板 / 约束从未被执行或复用 → 提议删除
- 新工作流模式值得固化 → 提议新增规则

Rune 是活文档，不是刻在石头上的。**该加就加，该砍就砍**——每条规则都必须能在实际项目里观测到收益，否则它就是噪音。

## 八、消费项目接入

新项目接入 Rune 的核心步骤（参考实现：SIGIL、Vates）：

1. **挂载 submodule**

   ```bash
   git submodule add <rune-url> .rune
   ```

   同 org 用相对路径 `../Rune`；跨 org 用绝对 URL。

2. **在消费项目 `CLAUDE.md` 导入 L1**

   ```markdown
   @.rune/rules.md
   ```

   Claude Code 原生 `@` 导入会把本文件纳入 session 上下文。

3. **部署自保护模板**（在消费项目自有路径，**不是**引用 Rune）

   最少 4 份资产：

   - `.claude/hooks/session-start.sh` — 失败探测 hook，`.rune/rules.md` 缺失时 auto-init；仍失败时向 stdout 打横幅（`stderr` 对 Claude 不可见）
   - `scripts/bootstrap.sh` — 一次性 dev setup，第一步永远是 `git submodule update --init --recursive`
   - `CLAUDE.md` — 项目 L2；含"与 Rune 的协同"章节 + 规则级约束表
   - `docs/rune-integration-troubleshooting.md` — 三层契约自检清单

   **关键约束**：这些资产**必须原样复制进消费项目的 git 历史**，不能软链或 `@` 引用回 Rune——Rune 缺席时，自保护逻辑不能跟着一起消失。

4. **初次 bootstrap + 自检**

   ```bash
   bash scripts/bootstrap.sh
   ```

   自检的唯一可信测试：**让 Claude 复述本文件规则 1 的四个字段**，能准确复述说明 `@` 导入真实生效。

### 升级 Rune pin

```bash
git submodule update --remote .rune
git add .rune && git commit -m "chore: bump Rune pin"
```

**独立 commit、独立 PR**。若本次 bump 带入新 L1 规则或修改现有规则，**同一 PR 必须更新消费项目的规则级约束表**——否则新规则生效但合规不可观测，等同静默失效。

## 九、反模式（接入与维护层面）

- ❌ 把本文件拷贝进消费项目仓库（漂移）
- ❌ 把项目专属规则搬进本文件（L2 留在项目本地）
- ❌ session-start hook 用 `stderr` 报告失败（对 Claude 不可见 → 静默失效）
- ❌ session-start hook 因 `.rune/` 缺失而 `exit` 非 0（block session）
- ❌ 规则注入逻辑放在环境判断之后（本地 session 拿不到规则）
- ❌ 依赖 `@` 导入的失败行为做失败检测（未文档化，可能静默——必须靠 hook banner 兜底）
- ❌ `.rune` pin bump 不在同一 PR 更新 L2 规则级约束表（新规则生效但合规不可观测）
- ❌ 自保护模板（hook / bootstrap / CLAUDE.md / troubleshooting）用符号链接或 `@` 引用回 Rune（Rune 缺席时这些也一起消失，消费项目失去自救能力）

---

## 十、反模式（工程层面）

- ❌ 类比思维：因为某框架/库这么做，所以我也这么做——必须追溯第一性原理
- ❌ 跳过五步工程法：还没质疑需求就动手实现
- ❌ 防御性冗余：在上游已保证的地方再写一遍校验
- ❌ 惯性变换：没明确目的的归一化 / 平滑 / 填充
- ❌ 靠感觉说"做完了"：无自动化验证、无 E2E 截图、无测试证据
- ❌ 用"以后可能用到"保留死代码 / 注释代码
- ❌ 用 mock 内部模块跑单元测试（等于测了个空气）
- ❌ 文档里写代码行号 / 私有符号 / 重复 dataclass 默认值
- ❌ 本地 CI 脚本超 90 秒（gate 会腐化，没人愿意等）
