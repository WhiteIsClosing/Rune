# Claude 工作流规则（来自 Rune）

以下规则由 SessionStart hook 注入到当前 session 上下文，对所有接入 Rune 的项目生效。

---

## 规则 1：先定验收标准，再动手

收到**任务类请求**时，**不要立即开始实现**。先输出一份验收标准，等用户确认或修改后再执行。

验收标准包含四个部分：

### 1. 验证方式
做完后如何证明对了？优先可自动化：
- 跑什么命令（测试、lint、type check、构建等）
- 看什么输出或断言
- 如果必须手动验证，写清具体步骤

### 2. 必须覆盖的边界
列 2-4 个最关键的 case。不追求穷举，只列"漏了会翻车"的。

### 3. 停下来找用户的信号
什么情况应该回来问，而不是自己猜？例如：
- 遇到多种合理的设计选项
- 发现原需求和现有代码冲突
- 需要引入新依赖 / 改动公共接口

### 4. 范围外的事
明确**不做**的东西，防 scope creep。

---

## 规则 2：不走验收标准的例外

以下情况直接动手，**不要**先输出验收标准：

- Prompt 中包含字符串 `skip criteria`
- 对话、探索、闲聊、概念讨论、纯问答类请求（用户在"聊"不是"派活"）
- 零风险只读操作：读文件、查询命令、跑状态检查等

判断不确定时，倾向于**先输出简短的验收标准**——用户总可以说"skip criteria"让你跳过，反之不成立。

---

## 规则 3：每完成一类任务，沉淀模板

完成任务后，如果该任务属于一个**可能再次出现的类别**（前端组件、SQL 迁移、API 接入、bug 修复等），主动建议用户：

> 「这类任务以后还会有。要不要把刚才的验收标准抽象成模板沉淀下来？」

用户同意后，把模板写成简短的 markdown 文件归档。下次遇到同类任务直接套模板，冷启动成本趋近于零。

---

## 元规则：持续打磨 Rune 本身

如果在协作过程中发现：
- 某条规则写得不清楚 / 会误判
- 有新的工作流模式值得固化
- 当前规则和实际项目冲突

**主动提议更新 `rules.md`**。Rune 是活文档，不是刻在石头上的。
